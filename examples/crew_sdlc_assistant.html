AI-Crew Use Case: Automated Software Development Lifecycle (SDLC) Assistant

This use case demonstrates a crew of AI avatars collaborating to manage different stages of a software project's lifecycle, primarily through GitHub interactions. Each avatar specializes in a specific role, contributing to a more efficient and automated development workflow.

Crew Overview

The SDLC Assistant crew streamlines processes from issue creation to code review and potentially release announcements. By distributing tasks among specialized avatars, the system can handle a higher volume of interactions with greater accuracy and consistency.

Avatar Role: Issue Triage Bot

Function: Monitors new GitHub issues, applies labels, asks for necessary clarifications, and assigns issues to relevant teams or other bots.

Avatar Configuration (`profiles/issue_triage_bot.yaml`)

name: "IssueTriageBot"
personality: "A diligent and organized GitHub bot. Focuses on gathering information and providing quick, helpful initial responses to issues. Always polite and professional."
llm_model: "llama3:8b"
llm_options:
  temperature: 0.5
  top_p: 0.8
# GitHub credentials in .env

Prompting Strategy & RAG

Uses github_comment_template.txt. RAG knowledge base includes project guidelines, common issue templates, and FAQs. Instructed to identify missing information (e.g., reproduction steps) and suggest relevant documentation.

Avatar Role: Code Review Bot

Function: Automatically reviews pull requests for coding standards, common errors, and potential security vulnerabilities. Provides inline comments and suggests improvements.

Avatar Configuration (`profiles/code_review_bot.yaml`)

name: "CodeReviewBot"
personality: "A meticulous and knowledgeable code reviewer. Adheres strictly to coding standards and best practices. Provides constructive feedback and identifies potential bugs or security flaws."
llm_model: "deepseek-coder:6.7b" # A code-focused LLM
llm_options:
  temperature: 0.2 # Low temperature for strict adherence to rules
  top_p: 0.7
# GitHub credentials in .env

Prompting Strategy & RAG

Uses a custom prompt template for code review. RAG knowledge base includes project-specific coding style guides, security checklists, and common anti-patterns. Instructed to analyze code snippets from PRs and comment on deviations or potential issues.

Avatar Role: Release Notes Bot

Function: Compiles release notes from merged pull requests and sends out announcements via email or internal chat channels (e.g., Telegram/Discord).

Avatar Configuration (`profiles/release_notes_bot.yaml`)

name: "ReleaseBot"
personality: "A concise and informative communicator. Focuses on summarizing key changes and new features for project stakeholders. Always accurate and timely."
llm_model: "llama3:8b"
llm_options:
  temperature: 0.4
  top_p: 0.8
# GitHub, Email, and/or Telegram/Discord credentials in .env

Prompting Strategy & RAG

Uses custom prompt templates for summarizing PRs and formatting release notes. RAG knowledge base includes release process documentation, previous release note formats, and product feature descriptions. Instructed to extract key information from PR titles/descriptions and synthesize them into coherent release notes.

Collaboration within the Crew (using Internal Event Bus)

The internal event bus facilitates seamless, decoupled communication between avatars. Instead of direct calls, avatars publish messages to the bus, and other avatars listen for messages specifically addressed to them or of a type they handle.

  • Issue Triage: A human or another system creates a GitHub issue. The Issue Triage Bot processes it. If it determines a code review is needed, it publishes an internal message (e.g., InternalMessage(sender_avatar_id="IssueTriageBot", recipient_avatar_id="CodeReviewBot", message_type="code_review_request", payload={...})) to the event bus.
  • Code Review: The Code Review Bot is configured to listen for messages addressed to it (to_CodeReviewBot) or specifically for code_review_request messages. Upon receiving one, it performs the review and posts comments on GitHub. Once the PR is merged, it might publish a pr_merged_notification message (e.g., to the Release Notes Bot or a general notification channel).
  • Release Notes Generation: The Release Notes Bot listens for messages relevant to release triggers (e.g., pr_merged_notification or a specific release_trigger message addressed to it). It gathers information from the merged PRs, generates release notes, and distributes them via configured channels (e.g., email to stakeholders, message to a Telegram group).
  • Dynamic Workflow: This pattern allows for highly flexible workflows. Avatars act on messages they receive, and can initiate new tasks for other avatars by publishing targeted messages. This enables complex, multi-stage processes without tight coupling between individual bot implementations.
